# PhoneVerse 产品需求与开发说明

> AI 手机模拟器引擎  
> 文档版本：v0.2  
> 当前产品版本：v0.1 MVP  
> 开发方式：Web First，后续可封装为 Android / iOS App

---

## 1. 项目目标

PhoneVerse 是一个运行在浏览器中的沉浸式手机模拟器。用户进入页面后，可完成解锁、打开应用、查看联系人、收发模拟消息和接收通知等操作。

v0.1 的目标不是实现完整手机系统，也不接入 AI，而是验证以下核心体验：

> 用户能够在一部视觉统一、交互流畅的模拟手机中完成“解锁 → 打开聊天 → 发送消息 → 收到本地模拟回复或通知”的完整流程。

项目后续可扩展为 AI 聊天、互动剧情、角色模板和模板商城。

---

## 2. 产品定位与运行形态

### 2.1 运行平台

v0.1 以 Web 为主：

- 桌面浏览器：页面中央显示完整手机容器。
- 移动浏览器：使用全屏布局，适配安全区域。
- 暂不安装 Capacitor，也不生成原生 App。
- 代码需保留平台适配层，避免未来封装 App 时重写核心业务。

### 2.2 视觉原则

采用 **Apple-inspired** 设计语言，但不得 1:1 复制 iOS：

- 不使用 Apple 商标、系统名称或受版权保护的系统图标。
- 不捆绑 SF Pro 字体文件，优先使用系统字体栈。
- 使用自有图标、Lucide Icons 或其他许可明确的图标库。
- UI 强调简洁、柔和、克制和沉浸感。

### 2.3 语言与主题

- v0.1 仅支持简体中文。
- v0.1 仅提供浅色主题。
- 颜色、圆角和间距必须使用 Design Tokens，便于后续增加深色模式与多语言。

## 3. 版本边界

### 3.1 v0.1 MVP 的定义

v0.1 MVP 包含 P0、P1、P2 三个里程碑，必须按顺序开发和验收。

| 里程碑 | 范围 | 目标 |
|---|---|---|
| P0 | 手机容器、锁屏、桌面、静态 Chat | 完成最小可演示闭环 |
| P1 | Contacts、会话数据、通知系统 | 建立核心交互与数据关系 |
| P2 | Photos、Notes、Settings、体验优化 | 完成可展示的 v0.1 MVP |

### 3.2 Chat 与 AI 的边界

- v0.1 Chat 仅使用本地数据与本地规则。
- 用户可以输入并发送文本消息。
- 联系人的回复由预设脚本或固定规则生成，不调用任何 LLM。
- AI 聊天属于 Phase 2，不在 v0.1 范围内。

### 3.3 后续阶段

| 阶段 | 内容 |
|---|---|
| Phase 2 | 引入 AI 聊天、角色 Prompt、轻量 BFF/API、密钥管理 |
| Phase 3 | 电话、相册上传、天气、音乐等扩展应用 |
| Phase 4 | 剧情系统、分支选择、多结局 |
| Phase 5 | 模板导入导出、模板商城、Capacitor 封装 |

Phase 2 必须使用后端代理转发 LLM 请求。禁止在前端保存或暴露商业 API 密钥。

## 4. 核心用户流程

### 4.1 主流程

1. 用户打开网页。
2. 页面显示锁屏、时间、日期、壁纸和一条预设通知。
3. 用户上滑或点击解锁区域。
4. 用户进入桌面。
5. 用户打开 Chat。
6. 用户进入“哥哥”的会话。
7. 用户发送一条文本消息。
8. 系统将消息保存到本地。
9. 本地脚本在 800–1500 毫秒后返回一条固定模拟回复。
10. 用户退出 Chat。
11. 本地事件引擎触发一条新消息。
12. 页面显示通知横幅。
13. 用户点击通知，进入对应会话。

### 4.2 通知触发规则

v0.1 的通知不依赖 AI，由本地事件引擎触发：

- 首次解锁后，可在 5 秒后触发一条预设消息。
- 用户不在目标会话时：新增消息、增加未读数并显示通知横幅。
- 用户正在目标会话时：仅追加消息，不显示通知横幅。
- 点击通知后：进入对应会话、标记通知已读、清除该会话未读数。
- 同一预设事件每个初始化周期只触发一次。
- Settings 中提供“重置演示数据”，用于重新体验完整流程。

## 5. 功能需求

## 5.1 P0：最小可演示版本

### 5.1.1 手机容器

桌面端：

- 推荐设计基准尺寸：390 × 844。
- 手机容器在页面中水平、垂直居中。
- 视口不足时按比例缩放，不出现横向滚动条。
- 手机外框、圆角、阴影和安全区域统一由 `PhoneShell` 管理。

移动端：

- 使用 `100vw × 100dvh`。
- 不强制显示手机外框。
- 适配顶部和底部安全区域。

### 5.1.2 锁屏

显示：

- 当前时间。
- 当前日期。
- 内置壁纸。
- 一条预设消息通知。
- 解锁提示。

交互：

- 支持上滑解锁。
- 为桌面端提供点击解锁作为替代操作。
- 锁屏状态刷新后可恢复。

### 5.1.3 桌面

首屏显示以下应用：

- Chat
- Contacts
- Photos
- Notes
- Settings

要求：

- 使用统一图标尺寸、圆角和间距。
- 点击应用后进入应用页面。
- 支持返回桌面。
- v0.1 不实现图标拖拽、文件夹、分页和长按菜单。

### 5.1.4 Chat

Chat 首页：

- 显示会话列表。
- 每项显示头像、昵称、最后一条消息、时间和未读数。

会话页面：

- 顶部显示返回按钮、头像、昵称和状态。
- 中间显示消息列表。
- 底部显示文本输入框和发送按钮。
- 支持发送非空文本消息。
- 发送后自动滚动到底部。
- 本地脚本返回固定模拟回复。
- 暂不支持图片、语音、撤回、回复引用和消息编辑。

## 5.2 P1：联系人与通知

### 5.2.1 Contacts

联系人列表至少包含：

- 哥哥
- 妈妈
- 妹妹
- 同学
- AI 助手

规则：

- 一个联系人对应一个默认会话。
- 点击联系人直接进入其会话。
- 联系人资料来自本地种子数据。
- v0.1 不支持新增、删除或编辑联系人。

### 5.2.2 Notification

通知系统包含：

- 通知横幅。
- 通知点击跳转。
- 未读数同步。
- 已读状态同步。
- 本地事件触发。

通知横幅字段：

- 应用图标。
- 联系人昵称。
- 消息摘要。
- 相对时间。

通知横幅行为：

- 默认显示 4 秒后自动收起。
- 用户点击后立即跳转。
- 同一时刻最多显示一条通知。
- 新通知到达时按先进先出顺序处理。

## 5.3 P2：辅助应用与体验优化

### 5.3.1 Photos

- 展示内置静态图片。
- 图片来源为项目自有或许可明确的资源。
- 使用网格布局。
- 点击后打开全屏预览。
- v0.1 不支持上传、删除和编辑。

### 5.3.2 Notes

- 支持创建、编辑和删除纯文本备忘录。
- 内容自动保存到 IndexedDB。
- Notes 与 Chat 暂无业务关联。
- 不支持富文本、附件、标签和协作。

### 5.3.3 Settings

包含：

- 产品名称。
- 当前版本。
- 关于说明。
- “重置演示数据”按钮。

重置操作需：

- 清空本地业务数据。
- 重新写入种子数据。
- 返回锁屏。
- 允许预设通知事件再次触发。

## 7. 状态与架构

### 7.1 状态划分

IndexedDB：

- contacts
- threads
- messages
- notifications
- photos
- notes
- settings

Zustand：

- 当前系统页面。
- 当前打开的应用。
- 当前会话。
- 锁屏状态。
- 当前通知横幅。
- 应用导航栈。
- 模态窗口状态。

### 7.2 导航方式

PhoneVerse 内部使用“应用栈 + 模态”的导航模型，不依赖浏览器 URL 表达每个手机页面。

要求：

- 系统层负责锁屏、桌面和应用切换。
- 应用层负责应用内部页面栈。
- 浏览器刷新后恢复到安全页面，默认可回到锁屏或桌面。
- v0.1 不要求浏览器前进/后退按钮映射到手机内部导航。

### 7.3 平台适配层

为未来 Capacitor 封装预留统一接口：

```ts
interface DeviceService {
  getPlatform(): "web" | "android" | "ios";
  showNativeNotification?(title: string, body: string): Promise<void>;
  pickImage?(): Promise<string | null>;
}
```

v0.1 只实现 Web Adapter，不安装原生插件。

## 9. UI 与交互规范

### 9.1 Design Tokens

至少统一定义：

- 颜色。
- 字体。
- 字号。
- 行高。
- 圆角。
- 间距。
- 阴影。
- 动画时长。
- Z-index 层级。

禁止在组件中大量散落硬编码视觉值。

### 9.2 动画

使用 Framer Motion 实现：

- 解锁。
- 应用打开与关闭。
- 页面切换。
- 消息出现。
- 通知横幅。
- 图片预览。

要求：

- 动画时长建议为 160–320 毫秒。
- 不使用大幅弹跳和夸张缩放。
- 支持 `prefers-reduced-motion`。
- 动画不得阻塞核心操作。

### 9.3 无障碍最低要求

- 所有可点击元素支持键盘聚焦。
- 图标按钮必须提供 `aria-label`。
- 表单必须具有关联标签或可访问名称。
- 文本与背景保持基本可读对比度。
- 桌面端可使用 Enter 完成解锁和发送消息。

## 12. 验收标准

### 12.1 P0 完成标准

- 桌面端和移动端均可正常显示。
- 用户可从锁屏进入桌面。

- 用户可进入“哥哥”会话并发送文本。
- 系统可返回一条本地固定回复。

- 本地事件可触发新消息。

- 支持键盘完成主要操作。

## 14. Cursor 执行指令

1. 严格按照 P0 → P1 → P2 的顺序开发。
2. 每次只实现一个明确任务，不得提前实现后续阶段功能。
3. 开始任务前，先列出：
   - 本次目标。
   - 预计修改或新增的文件。
   - 关键实现方案。
4. 完成任务后，必须：
   - 说明实际修改内容。
   - 运行相关测试。
   - 报告未完成项和已知问题。
5. 组件保持单一职责，避免超长文件和重复代码。
6. 数据访问必须通过统一 Repository 或 Service，不得在 UI 组件中直接操作 IndexedDB。
7. 业务逻辑不得写入纯展示组件。
8. 不得在前端写入任何 API 密钥。
9. 不得使用来源或许可不明确的图片、字体和图标。
10. 遇到未定义细节时，优先采用最小实现，不得扩大范围。
11. 若需求与本文件冲突，以本文件的“版本边界”和“明确不做”为准。
12. 每个里程碑通过验收后，再开始下一里程碑。

## 15. 商业化与后续架构要求

- Phase 2 增加轻量 BFF/API，仅负责模型请求转发、鉴权、额度控制和密钥管理。
- AI Provider 通过统一接口接入，避免业务代码绑定单一模型。
- 用户模板与角色配置应可序列化，便于后续导入、导出和售卖。
- Web 与原生能力通过 Adapter 隔离。
- 商业版本发布前需复核第三方依赖、字体、图标和图片许可。
